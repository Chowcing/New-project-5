# 记录详情页重设计方案

日期：2026-06-20

## 背景

当前记录详情页已经支持查看、编辑、复制为今日、设为周期、删除、凭证图片预览与编辑态图片管理。页面的问题主要在于查看态信息重复、首屏操作区占位偏大、编辑态像从详情页跳到独立表单，和“记一笔”的视觉语言没有形成顺畅延续。

本次重设计采用用户确认的“账单脉络型”方向：让用户像读一张账单一样快速理解这笔记录的发生时间、分类、支付方式、渠道地点和备注，同时让编辑态沿用同一顺序，降低查看和修改之间的割裂感。

## 目标

- 查看态一眼看清金额、事项、时间、分类、支付方式、渠道地点、备注和凭证状态。
- 视觉质感比现状更完整，延续现有 iOS 科技风、玻璃卡片和主题 token。
- 编辑态与查看态共用信息顺序，并和“记一笔”的高频录入结构保持一致。
- 保留现有业务能力和接口，不新增后端字段或接口。

## 非目标

- 不改交易接口、数据库结构、图片上传规则或 OCR 流程。
- 不引入新的组件库或独立视觉体系。
- 不重做流水列表页、记一笔页或全局导航。
- 不新增同类交易统计、审计信息、地图展示等需要后端支持的能力。

## 查看态设计

页面结构调整为四个区域：摘要卡、账单脉络、凭证图片、操作区。

摘要卡放在首屏最上方，突出事项标题和金额。标题使用 `transactionTitle(record)`，金额按收入和支出展示 `+¥` 或 `-¥`，颜色继续使用 `--income` 和 `--expense`。摘要卡同时展示类型、渠道、中文友好时间、分类、支付方式和地点标签，让用户不进入详情区也能快速确认这笔记录。

账单脉络区采用纵向列表或轻量时间线样式，顺序固定为：

1. 发生时间
2. 分类
3. 支付方式
4. 渠道与地点或线上平台
5. 备注

每一项使用图标加文本，保持移动端可扫读。线上记录显示“线上平台”，线下记录显示“线下地点”。备注为空时显示“无备注”，但保持在同一位置，避免布局跳动。

凭证图片区继续使用现有 `record.images` 和鉴权图片接口加载缩略图。无图片时可以不渲染图片区，避免空面板挤占空间；有图片时显示数量提示和三列缩略图，点击继续调用现有图片预览。

操作区从首屏中段下移到内容底部，保留四个操作：编辑记录、复制为今日、设为周期、删除记录。编辑作为主按钮，其余为次按钮，删除使用危险样式。这样首屏优先读信息，不被操作按钮打断。

## 编辑态设计

编辑态沿用查看态的信息顺序，但把脉络项变为可编辑字段。整体结构为：编辑摘要卡、账单脉络编辑区、凭证图片管理区、底部保存栏。

编辑摘要卡包含类型切换、金额输入和事项输入。金额继续带人民币符号视觉提示，表单值仍只保存数字金额。类型切换沿用现有支出/收入逻辑，类型变化时同步分类候选。

账单脉络编辑区按查看态顺序排列：

1. 时间：继续使用 `ModernDateField`，展示和值格式保持现有提交逻辑。
2. 分类：保留快捷 chip 和 `BottomSheet` 更多选择能力。
3. 支付方式：保留快捷 chip 和 `BottomSheet` 更多选择能力。
4. 渠道与地点：保留线上/线下切换；线上继续编辑 `onlineApp`，线下继续使用 `AmapPlaceField`。
5. 备注：使用现有 `van-field`。

凭证图片管理区保留现有能力：展示已有凭证、预览、单张删除、追加上传、最多 3 张、单张 5MB、类型校验和重复业务规则不变。保存仍先更新字段，再追加图片；图片上传失败时保留现有“记录已更新，凭证保存失败”提示。

底部保存栏继续使用 `FormActionBar`，保证安全区和移动端固定操作体验一致。取消编辑由导航返回触发，行为保持现有 `cancelEdit()`。

## 数据和状态

本次只使用 `TransactionRecord` 现有字段：

- `type`
- `itemName`
- `amount`
- `occurredAt`
- `channel`
- `onlineApp`
- `offlinePlace`
- `paymentMethodId`
- `paymentMethodName`
- `categoryId`
- `categoryName`
- `categoryIcon`
- `note`
- `images`

查看态从 `record` 派生展示文本，不新增请求。编辑态继续使用现有 `form`、`fillForm()`、`transactionPayload()`、`submit()`、`loadOptions()` 和图片加载状态。

需要新增或调整的展示辅助计算包括：

- 中文友好时间展示，例如 `2026年06月20日 12:24`。
- 摘要标签列表，例如类型、渠道、分类、支付方式、地点、凭证数量。
- 脉络项列表，按记录字段派生 label、icon、value 和辅助说明。

## 组件和样式

实现优先留在 `frontend/src/views/TransactionDetailView.vue`，因为本次是单页重设计，暂不抽新组件。若实现过程中发现查看态和编辑态脉络项重复明显，可在同文件内用小型数据结构驱动模板，不新增跨页面抽象。

样式必须复用 `frontend/src/styles/main.css` 和 `frontend/src/utils/themes.ts` 的 token。允许新增页面内语义类，但不新增裸色、孤立字号或随意圆角。按钮和信息行继续使用“图标 + 文本”，纯图标按钮必须保留可访问名称。

弹窗继续使用 `BottomSheet`。页面内不新增直接 `<van-popup>`。

## 错误处理和边界

- 详情加载失败保持现有 `showError(error, '记录详情加载失败')` 和空态。
- 图片加载中、部分失败、预览为空时保持现有状态处理。
- 编辑态没有字段变化且没有新增图片时继续提示“没有修改内容”。
- 分类或支付方式未加载、金额非法、缺少时间、线下缺地点、线上支出缺 APP 时保持现有校验。
- 删除记录和删除图片继续二次确认。
- 320px 宽度下摘要金额、标题、标签、脉络值必须换行或收缩，不允许横向溢出。

## 验证计划

前端改动完成后至少执行：

```bash
cd frontend
npm run build
```

`npm run build` 会先执行 `npm run check:ui`，覆盖 UI token、直接 `van-popup` 等约束。实现涉及移动端详情页首屏和布局，完成后还需要人工检查窄屏、浅色主题、深色主题、收入记录、支出记录、线上记录、线下记录、有图片和无图片状态。

## 交付标准

- `/records/:id` 查看态使用新的“摘要卡 + 账单脉络 + 凭证图片 + 操作区”结构。
- 编辑态和查看态信息顺序一致，且保留现有编辑、保存、取消和图片管理能力。
- 日期时间面向用户显示为中文友好格式，不再直接展示 `datetime-local` 风格的 `T` 分隔格式。
- 不改变后端接口、不改变交易 payload、不改变图片上传约束。
- 前端构建通过。
